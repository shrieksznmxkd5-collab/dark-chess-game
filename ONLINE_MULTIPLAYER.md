# 《暗棋大乱斗》网页在线对战开发要求

请在现有单机、人机和本地双人玩法不被破坏的前提下，为项目增加网页在线双人对战。

## 一、技术架构

使用：

* Netlify：托管前端网页；
* Supabase Auth：匿名玩家身份；
* Supabase Database：房间、玩家和棋局状态；
* Supabase Realtime：房间在线状态和棋局更新；
* Supabase Edge Functions：创建房间、加入房间、初始化棋局和验证移动。

第一版不要求用户注册邮箱。玩家进入网站后使用匿名登录，并输入临时昵称。

网页中只能使用 Supabase URL 和 Publishable Key。

严禁把 Secret Key 或 service_role key 写入前端代码、GitHub仓库或公开配置文件。

## 二、重要安全规则

本游戏的暗棋包含隐藏的 `realType` 和 `realCamp`。

在线模式下：

1. 完整暗棋身份只能保存在服务端私有棋局状态中；
2. 未翻面的棋子不得向客户端发送 `realType` 和 `realCamp`；
3. 客户端只能获得公开棋局快照；
4. 玩家只能提交移动意图，不能提交新的完整棋局状态；
5. 服务端必须验证移动、吃子、翻面、阵营转换、将帅照面和延迟威胁；
6. 服务端验证成功后，才更新公开棋局状态；
7. 不允许玩家通过浏览器开发者工具提前查看暗棋身份。

## 三、首页模式

首页增加：

1. 人机对战；
2. 本地双人；
3. 创建好友房；
4. 加入房间；
5. 快速匹配；
6. 游戏规则。

现有人机模式和本地双人模式继续离线运行，不依赖Supabase。

## 四、匿名玩家

玩家第一次使用在线模式时：

1. 调用Supabase匿名登录；
2. 获取唯一用户ID；
3. 要求输入昵称；
4. 昵称保存在本地和房间成员记录中；
5. 刷新网页后尽量恢复当前会话。

## 五、创建好友房

玩家点击“创建好友房”后：

1. 服务端创建房间；
2. 生成唯一六位房间码；
3. 创建者成为房主；
4. 默认房主为红方；
5. 房间状态为 `waiting`；
6. 返回邀请链接，例如 `?room=AB7K2Q`；
7. 房主进入房间大厅。

房间大厅显示：

* 房间码；
* 复制房间码；
* 复制邀请链接；
* 红方玩家；
* 黑方玩家；
* 双方准备状态；
* 离开房间按钮。

## 六、加入好友房

玩家可以：

1. 输入六位房间码；
2. 或打开包含房间码的邀请链接；
3. 输入昵称；
4. 加入未满且仍处于waiting状态的房间；
5. 成为黑方玩家。

需要处理：

* 房间不存在；
* 房间已满；
* 房间已经开始；
* 房间已过期；
* 玩家重复加入。

## 七、准备与开始

1. 两名玩家都进入后可以点击准备；
2. 双方都准备后，由服务端初始化棋局；
3. 服务端完成暗棋随机洗牌；
4. 完整状态写入私有棋局；
5. 去除未翻面棋子真实身份后生成公开快照；
6. 房间状态变为 `playing`；
7. 双方同时进入棋局页；
8. 红方先行动。

暗棋随机结果只能由服务端生成。

## 八、在线移动

客户端发送：

```js
{
  roomId,
  expectedVersion,
  from: { row, col },
  to: { row, col }
}
```

服务端需要验证：

1. 玩家是否属于该房间；
2. 是否轮到该玩家；
3. 该棋子当前是否归玩家控制；
4. 基础移动是否合法；
5. 吃子是否合法；
6. 暗棋首次行动是否需要翻面；
7. 翻面后阵营和控制权是否正确；
8. 将帅是否照面；
9. 普通将帅安全；
10. 延迟威胁状态机；
11. 将帅是否被合法吃掉；
12. `expectedVersion` 是否与服务端一致。

验证成功后：

1. 更新私有棋局状态；
2. 生成新的公开快照；
3. 棋局版本加一；
4. 写入移动记录；
5. 通过Realtime通知双方刷新。

验证失败时：

1. 不修改棋局；
2. 返回明确错误原因；
3. 客户端重新读取最新公开快照。

## 九、公开与私有棋局

至少设计：

* `game_states_private`：完整暗棋身份，仅服务端访问；
* `game_snapshots_public`：去除未翻面真实身份，房间成员可读取；
* `moves`：保存着法顺序和结果。

不要让普通客户端获得私有棋局表的读取权限。

## 十、快速匹配

点击快速匹配后：

1. 玩家进入 `match_queue`；
2. 系统按加入时间匹配最早的两名玩家；
3. 使用事务或数据库函数避免重复匹配；
4. 创建新房间；
5. 随机分配红黑阵营；
6. 两人自动进入房间；
7. 初始化棋局并开始；
8. 玩家可以取消匹配。

第一版不需要积分或段位匹配。

## 十一、实时状态

使用Realtime同步：

* 玩家加入；
* 玩家离开；
* 准备状态；
* 对手在线状态；
* 房间状态；
* 新棋局快照；
* 游戏结束；
* 再来一局邀请。

棋局最终状态以服务端数据库为准，不以客户端消息为准。

## 十二、断线恢复

1. 玩家暂时断线时显示“对手暂时离线”；
2. 保留房间和棋局状态；
3. 玩家重新打开链接后恢复棋局；
4. 重新读取最新公开快照和棋局版本；
5. 在线状态恢复后继续游戏；
6. 第一版可设置60至120秒重连提示；
7. 超时后允许另一方离开房间，但不自动修改棋盘胜负。

## 十三、数据库结构

至少需要：

* `rooms`
* `room_members`
* `game_states_private`
* `game_snapshots_public`
* `moves`
* `match_queue`

请生成SQL迁移文件，并为公开表配置RLS。

只有房间成员能读取对应的公开房间和棋局数据。

私有棋局只能由服务端函数访问。

## 十四、代码结构

不要把联网代码混进现有规则文件。

建议增加：

```text
js/network/
├─ supabase-client.js
├─ auth-service.js
├─ room-service.js
├─ matchmaking-service.js
├─ realtime-service.js
└─ online-game-controller.js

supabase/
├─ migrations/
└─ functions/
   ├─ create-room/
   ├─ join-room/
   ├─ set-ready/
   ├─ start-game/
   ├─ submit-move/
   ├─ join-matchmaking/
   └─ leave-room/
```

现有游戏规则应重构为可在浏览器和服务端复用的纯JavaScript模块：

```js
applyMove(gameState, move)
getLegalMoves(gameState, player)
createPublicSnapshot(privateState)
```

规则函数不能直接操作DOM。

## 十五、开发顺序

第一阶段：

1. 检查现有游戏规则是否已与UI分离；
2. 把游戏引擎重构为可序列化、可在服务端复用；
3. 保证本地模式不受影响。

第二阶段：

1. 匿名登录；
2. 创建房间；
3. 房间码；
4. 邀请链接；
5. 加入房间；
6. 准备大厅。

第三阶段：

1. 服务端初始化棋局；
2. 私有和公开状态；
3. 在线提交移动；
4. Realtime同步；
5. 断线恢复。

第四阶段：

1. 快速匹配；
2. 取消匹配；
3. 再来一局；
4. 异常处理。

开发前先用中文说明：

1. 当前项目是否适合直接加入联网；
2. 哪些规则代码需要先重构；
3. 需要创建哪些Supabase资源；
4. 哪些步骤必须由用户在Supabase后台手动完成。

不要一次性重写整个项目。每完成一个阶段，先说明修改文件、测试方法和未完成事项。
